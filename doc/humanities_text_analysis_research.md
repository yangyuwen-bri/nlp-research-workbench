# 文科生文本分析场景调研与功能规划

## 1. 目的

这份文档用于帮助负责开发的 coding agent 或开发同学理解：

- 文科生在文本分析中的真实使用场景
- 这些场景会反推出什么产品能力
- 当前平台已经覆盖了哪些能力
- 哪些能力仍然缺失
- 前端应该如何围绕这些能力做简洁、可用的模块化改造

本文件重点是“以现有 NLP 分析平台为基础做合理收敛”，不讨论采集模块，不要求推倒重来。

---

## 2. 调研问题

本轮调研主要回答两个问题：

1. 文科生在做文本分析时，通常会做哪些事？
2. 这些行为应该如何转译成平台功能模块？

调研方法：

- 查阅数字人文、语料库分析、主题分析、文本工作坊相关论文与教学材料
- 从研究方法和论文产出反推用户需求

---

## 3. 结论摘要

### 3.1 总体结论

文科生通常不会把文本分析当成一个纯技术流程，而是把它当成研究工作的一部分。  
他们真正要完成的是：

- 整理语料
- 初步探索文本结构
- 找到可解释的模式
- 回看原文证据
- 形成论文或报告中的论点

所以他们对平台的期待，不是“模型越多越好”，而是：

- 能快速导入和预览语料
- 能看到词频、共现、主题、情感、分类等基础结果
- 能把结果和原文证据联动起来
- 能方便导出为论文或报告可用的中间成果

### 3.2 一个典型工作流

文科生/研究用户常见工作流大致如下：

1. 导入一批文本语料
2. 先看样本和字段，确认语料是否正确
3. 先做词频、选词、关键词、短语、上下文之类的探索
4. 看共词/共现、概念关系、主题结构
5. 再看情感、分类、主题聚类等更高层结果
6. 对重要结果回看原文片段
7. 写出解释性摘要、报告、论文附件或答辩材料

这意味着平台最重要的不是单个算法，而是“研究动作之间的串联”。

## 5. 反推出来的功能模块清单

根据调研，文科生常见需求可以抽象为以下 7 组模块。

### 5.1 语料管理

- 导入文本
- 预览样本
- 查看数据集元信息
- 基础清洗、去重、停用词控制
- 按来源/时间/作者/主题切分子语料

### 5.2 探索阅读

- 词频
- 关键词
- 短语/搭配
- 词的分布趋势
- 上下文检索 `KWIC / concordance`

### 5.3 关系发现

- 共词/共现
- 关系矩阵
- 网络关系
- 概念或人物关联

### 5.4 主题与标签

- 主题聚类
- 人工命名主题
- 分类标签
- 情感/立场辅助判断

### 5.5 证据联读

- 从词频回看原文
- 从主题回看代表片段
- 从分类/情感结果回看证据
- 原文与分析结果双向联动

### 5.6 解释与写作

- 自动摘要
- 研究报告草稿
- 主题说明
- 人工备注/修订

### 5.7 结果导出

- 词频表
- 匹配表
- 矩阵表
- 共词结果
- 分类/情感结果
- 主题摘要
- Markdown / Excel / CSV 导出

## 6. 当前平台覆盖对照

下表用于帮助开发者理解“哪些已有，哪些没有”。

| 模块 | 用户需求 | 当前平台覆盖情况 | 说明 |
|---|---|---|---|
| 语料导入 | CSV/XLSX/JSONL 导入 | 已覆盖 | 已有上传与解析 |
| 样本预览 | 看原文样本 | 已覆盖 | 现有前端已有 |
| 元信息展示 | 文件名、样本量、文本列 | 已覆盖 | 基础够用 |
| 子语料切分 | 按来源/时间/作者比较 | 未覆盖 | 文科研究高频需求 |
| 清洗控制 | 停用词、字段整理、去重 | 部分覆盖 | 有基础校验和去重，缺少前台控制 |
| 词频 | 高频词查看 | 已覆盖 | 当前已有 |
| 关键词 | 相对重要词而不只是高频 | 部分覆盖 | 现阶段更接近“选词” |
| 短语/搭配 | 常见词组、固定表达 | 未覆盖 | 需要补 |
| 趋势分布 | 某词在时间/文本中的分布 | 未覆盖 | 需要补 |
| 上下文检索 | KWIC、回看语境 | 未覆盖 | 非常重要的缺口 |
| 选词 | 人工相关词筛选 | 已覆盖 | 当前已有 |
| 匹配表 | 文本命中哪些词 | 已覆盖 | 当前已有 |
| 0/1 矩阵 | 结构化分析结果 | 已覆盖 | 当前已有 |
| 共词/共现 | 概念关联 | 已覆盖 | 当前已有共词边 |
| 网络可视化 | 直观看关系结构 | 部分覆盖 | 有数据，无强前端表达 |
| 情感分析 | 倾向判断 | 已覆盖 | 当前已有 |
| 文本分类 | 标签归类 | 已覆盖 | 当前已有 |
| 主题聚类 | 发现讨论主题 | 已覆盖 | 当前已有 |
| 主题人工修订 | 更贴近研究语言 | 部分覆盖 | 有 LLM 命名，缺少人工修订闭环 |
| 证据联读 | 点结果回原文 | 部分覆盖 | 有 snippet，但联动弱 |
| 人工备注/修正 | 研究者记录和修订 | 未覆盖 | 建议后续补 |
| 报告生成 | 形成阶段性摘要 | 已覆盖 | 当前已有 Markdown 报告 |
| 结果导出 | 论文/PPT/附录使用 | 已覆盖 | 当前较强 |
| 多次运行比较 | 比较不同运行结果 | 部分覆盖 | 有历史，但缺少比较视图 |

## 7. 当前平台的真实优势

从调研角度看，当前平台已经具备一个比较好的“基础研究分析台”雏形，尤其强在：

- 词频、选词、匹配、矩阵
- 共词关系
- 情感/分类/主题
- 报告与多格式导出

这意味着现阶段不需要增加太多新算法，也不应该急着扩到无关方向。  
更合理的做法，是围绕当前已有能力，把使用路径做得更清晰。

---

## 8. 最值得优先补强的能力

如果从文科生真实使用角度排序，以下能力优先级最高：

### 高优先

1. 子语料切分与比较
2. 上下文检索 `KWIC`
3. 从结果回跳原文

### 中优先

4. 短语/搭配分析
5. 多次运行结果比较
6. 主题人工修订

### 后续增强

7. 研究备注
8. 人工标签纠偏
9. 协作式代码本/主题本

## 9. 前端开发意图

### 9.1 当前问题

当前前端已经能展示很多结果，但整体看起来比较杂。  
主要原因不是功能太多，而是：

- 左侧堆叠了太多不同职责的面板
- 顶部 tab 和左侧信息区都在抢注意力
- 不同分析结果缺少清晰、统一的访问方式

### 9.2 想要达到的效果

前端不需要复杂设计，而应该尽量简洁、稳定、低认知负担。  
从用户使用角度，推荐改造成：

- 左侧：模块选择区
- 右侧：结果展示区

左侧只保留：

- 当前数据集选择
- 分析模块导航

右侧专门展示当前模块的内容。

建议模块至少包括：

- 总览
- 词频
- 情感
- 分类
- 主题
- 报告
- 导出

如果未来继续扩展，再加入：

- 选词
- 匹配
- 矩阵
- 共词

---

## 10. 前端开发方案建议

### 10.1 第一阶段：先做结构收敛

目标：

- 不改后端
- 不改功能逻辑
- 只重组前端布局和模块访问方式

建议：

1. 把顶部 tab 导航改为左侧模块导航
2. 左侧减少杂项卡片
3. 右侧作为唯一主展示区域
4. 每次只突出一个模块结果

### 10.2 第二阶段：补强研究动作

在现有结果基础上补：

1. 点词回看原文片段
2. 点主题看代表性文本
3. 情感/分类结果的证据联读

### 10.3 第三阶段：补研究增强能力

后续再考虑：

1. 子语料对比
2. KWIC 上下文视图
3. 人工修订/备注

---

## 11. 推荐给 coding agent 的实现原则

如果 coding agent 负责继续前端开发，建议遵守：

1. 不改后端接口，优先复用现有数据获取逻辑
2. 优先收敛页面结构，不先增加新功能
3. 保持设计简单、克制、易扫读
4. 左侧只保留“选择”，右侧只保留“展示”
5. 尽量避免重复导航
6. 若需要重构前端，优先拆分为小组件，避免继续把所有逻辑塞进单文件

---

## 12. 一句话总结

当前平台不是功能不够，而是“功能已经有了，但还没有按照文科研究者的使用路径组织好”。  
下一步最合理的方向，不是继续堆算法，而是围绕现有 NLP 模块，把平台收敛成一个更清晰的研究工作台。
